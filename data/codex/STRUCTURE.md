# The Codex Structure

Per Matt 2026-05-16: **The codex is what the engine produces.**

The codex is not a theology authored. The codex is the chosen body, walked by the engine, indexed and cross-referenced. The engine's job is binding + indexing — not synthesis. Matt curates the body; the engine surfaces it.

### Layer 3 — THE INDEX (navigational surfaces)

How readers find what's in the body. NOT authored chapters — pure index surfaces, generated by the engine from the body. The codex's "table of contents" is multiple indexes, not a doctrinal synthesis.

| Index | What it surfaces |
|---|---|
| **Scripture index** | Every verse, every cross-reference site across the body |
| **Person index** | Easton's persons + every textual appearance across the body |
| **Place index** | Easton's places + every textual appearance |
| **Theme index** | Emergent themes (Hebrews 11 / Faith Without Map → all sites; "the keeping" → all sites; "come and see" → all sites) drawn from cross-references, not authored |
| **Authority-tier index** | Every Words-in-Red passage; every Tier 3 patristic excerpt; every Tier 4 lens moment |
| **Author/work index** | Each work as a section, with chapters/verses listable directly |
| **Time index** | Chronological — works ordered by composition date when applicable |

Indexes are *navigational*, generated by the engine from the body — not authored content.

## What the codex compiler does

1. Walks Layer 1 (the body) — verifies all source files present, version-tagged
2. Walks Layer 2 (engine outputs) — collects verified-claim entries, cross-reference graph data
3. Generates Layer 3 (indexes) — computes scripture/person/place/theme indexes from the body + Layer 2 data
4. Emits three faces:
   - **Live URLs** at `/codex/<work>/<chapter>/<verse>` and `/codex/index/<index_name>/<key>`
   - **Signed artifact** at `compiled/codex_vN_M.{pdf,epub,json,sig}`
   - **Programmatic API** via the existing `/index/...` endpoints, extended

5. Signs everything with Ed25519. The signature certifies "this is the codex as of this date, with this body, walked by this engine version."

## What's NOT in the codex

By design, these don't belong in the codex layer:

- **Synthetic doctrinal chapters authored by Claude or the engine** — synthesis isn't what the engine produces. Indexes surface; they don't synthesize.
- **The engine's source code or architecture documentation** — that's `/about` material
- **Biographical material about M.R. Harris** — outside the codex's scope
- **The acquisition catalog or library inventory** — different layer (production substrate, not codex substrate)
- **Channel-specific content (Dade audio, etc.)** — those are downstream productions OF the substrate, not part of the codex itself

## Standing test for what enters the codex

> Does this belong in the chosen body of texts (Bible + named patristics + Matt's books)?
> Or is this engine-produced analysis of that body?
> Or is this a navigational index into that body?

If yes to any of those three: it's in the codex. If it's anything else, it lives elsewhere in the project.
