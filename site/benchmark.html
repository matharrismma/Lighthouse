Does the engine actually help?

What the benchmark measures

For each item, the same prompt is sent to the model in two configurations:

  • Alone: just the model. No tools.
  • With tools: three concordance-engine tools available — verify_chemistry, verify_physics_dimensional, verify_statistics_pvalue.

The model's reply is parsed (yes/no for classification items, decimal number for numeric items) and scored against the engine-derived ground truth. The discriminator is the per-domain delta: where do the tools change the answer?

The 37 items, by domain

Chemistry — 17

"Is this equation balanced?" Mix of balanced and unbalanced. Includes redox, decomposition, and combustion equations. The unbalanced ones are picked specifically to look balanced at a glance.

Statistics — 10

"Compute the two-tailed p-value." Two-sample t (3), paired t (3), one-proportion z (2), Fisher exact (2). Inputs designed to reach the model as plain text — exactly the format LLMs choke on without computational help.

Physics — 10

"Is this equation dimensionally consistent?" Mix of obviously-correct (F=ma), obviously-wrong (F=mv), and subtly-wrong (s = u·t² instead of s = u·t + ½a·t²).

Predictions, on the record

Before running the benchmark, here's the prediction:

Statistics: Claude alone scores below 50%. With tools: above 90%. This is the headline if it lands. p-value arithmetic from raw inputs is exactly what LLMs hallucinate confidently.

Chemistry: Claude alone scores around 75-85%. With tools: above 95%. The simple combustion equations are easy; the redox half-reactions and the "looks balanced but isn't" cases are the discriminators.

Physics: Both modes score similar, around 80-90%. Dimensional analysis is something modern LLMs handle reasonably well from training. The engine adds value mainly on the subtly-wrong cases.

If statistics shows the predicted gap, the engine has earned its keep. If it doesn't, that's a real finding — either the items are too easy, or LLMs have improved at p-value arithmetic faster than we thought.

Reproducing the benchmark

git clone https://github.com/matharrismma/Lighthouse
cd Lighthouse
pip install -e ".[mcp]" anthropic

export ANTHROPIC_API_KEY=sk-ant-...
python eval/benchmark/build_items.py            # rebuild items.jsonl
python eval/benchmark/adapter_anthropic.py      # ~30s, ~$0.05 on Haiku 4.5

The script writes results_*_alone.jsonl and results_*_tools.jsonl alongside printed summaries. Comparing the two is one diff away from "items the engine fixed" / "items the engine broke."

What this benchmark cannot prove

37 items is a starter benchmark, not a definitive one. It cannot:

  • Prove the engine helps on tasks outside the three domains tested.
  • Prove the engine helps with frontier models (GPT-5, Opus 4) that may already do mental p-value arithmetic better than Haiku.
  • Prove the engine helps in agentic workflows where tool latency matters.
  • Prove the engine catches errors in real published papers — that's a separate work item.

It can only prove what 37 items can prove: there is a measurable per-domain delta on a benchmark we authored, on a model we picked, in a configuration we controlled. That's a starting point, not a finish line.
